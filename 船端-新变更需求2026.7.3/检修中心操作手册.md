## 1. 系统总览与导航

### 1.1 应用路由

应用采用 Vue Router，共 5 条路由：

| 路由路径 | 名称 | 图标 | 说明 |
|---------|------|------|------|
| `/` | 首页 | view-dashboard | 全船设备总览仪表盘 |
| `/monitor` | 监控中心 | gauge | 实时设备监控、传感器数据 |
| `/repair` | **检修中心** | wrench | **本文档核心** |
| `/knowledge` | 知识库 | bookshelf | 设备维护手册、历史案例 |
| `/report/:id` | 机舱巡检日检报告 | clipboard-text | 巡检报告详情 |

顶部导航栏 `AppHeader` 始终展示前三个入口（首页 / 监控中心 / 检修中心），并显示系统时间和通知铃铛。

### 1.2 检修中心定位

检修中心是整个应用的**事件处置闭环核心**。它接收来自三个渠道的事件：
- **AI 检测**（`ai`）—— 监控中心传感器数据触发自动告警
- **巡检发现**（`inspection`）—— 机舱巡检日检报告中发现异常
- **船员上报**（`crew`）—— 船员通过异常上报弹窗手动提交

每个事件经过「确认 → 排查 → 维修 → 归档」的完整生命周期，最终关闭或标记误报。

## 2. 检修中心页面布局

检修中心路径 `/repair`，采用**左右分栏 + 全屏切换**布局。

```
┌─────────────────────────────────────────────────────────┐
│  [搜索框: 搜索事件标题、设备、描述...]      [检修中心]      │  ← 顶部栏
├──────────────┬──────────────────────────────────────────┤
│  筛选器       │  事件详情区                                │
│  ├ 设备系统   │  ┌────────────────────────────────────┐  │
│  ├ 优先级     │  │ 事件头部(标题/设备/标签/来源/时间)    │  │
│  ├ 状态       │  │ [AI分析] [排查维修] [操作日志]      │  │  ← Tab
│  └ [清空]     │  │                                    │  │
│              │  │  状态操作按钮                        │  │
│  事件列表     │  │  ─────────────────────────         │  │
│  ┌──────────┐│  │  Tab 内容区                         │  │
│  │ 事件卡片1 ││  │                                    │  │
│  │ 事件卡片2 ││  │                                    │  │
│  │ ...      ││  └────────────────────────────────────┘  │
│  └──────────┘│                                          │
└──────────────┴──────────────────────────────────────────┘
```

### 2.1 显示模式

| 模式 | 标识 | 触发 | 效果 |
|------|------|------|------|
| `split`（分栏） | `u.value === "split"` | 默认 / 点击「收起」按钮 | 左右分栏，列表 + 详情并排 |
| `fullscreen`（全屏） | `u.value === "fullscreen"` | 点击事件卡片 / 选中事件 | 侧边栏折叠，详情区占满全宽 |

点击事件卡片时调用 `Pt(event)`：
- 设置 `selectedEventId = event.id`
- 调用 store 的 `selectEvent(id)`
- 自动切换到 `fullscreen` 模式（除非通过 URL 参数 `eventId` 直接进入，则保持 `split`）

### 2.2 URL 参数

检修中心支持两个 query 参数：
- `eventId` —— 打开时自动选中并定位到指定事件
- `statusFilter` —— 打开时自动设置状态筛选条件

路由变更时通过 `watch` 监听，自动选中对应事件。

### 4.5 状态流转图

```
                         ┌──────────────────────────────────────────┐
                         │                                          ▼
  [创建事件] ──► pending ──confirm──► processing ──to_repair──► processing
                   │                    │  (stage 0→1)   (stage 1→2)
                   │                    │
                   │               temp_handle               temp_handle
                   │                    │                       │
                   │                    ▼                       ▼
                   │              temp_handled ◄─────────────────┘
                   │                    │
                   ├────false_alarm────►│
                   │                    │
                   └──────────────► resolve ◄──── from any state
                                       │
                                  (设备→NORMAL)
```

### 4.6 状态流转操作详解

状态流转通过 store 的 `updateEvent(eventId, action, options)` 函数实现。`action` 取值及效果：

| action | 中文名 | 前置状态 | 后置状态 | stageIndex 变化 | 日志阶段 | 日志标题 | progressSummary | 设备状态 |
|--------|--------|---------|---------|----------------|---------|---------|-----------------|---------|
| `confirm` | 确认事件 | pending | processing | 0 → 1 | 确认 | 事件已确认，进入排查 | 排查中：待执行排查项 | 不变 |
| `to_repair` | 进入维修 | processing | processing | 1 → 2 | 维修 | 排查完成，进入维修阶段 | 维修中：待执行维修项 | 不变 |
| `temp_handle` | 临时处理 | 任意 | temp_handled | → 2 | 维修 | 临时处理：{note} | 临时处理：待彻底排查/复装 | 不变 |
| `resolve` | 标记解决 | 任意 | resolved | → 2 | 归档 | 事件已解决并归档：{note} | 已归档：事件闭环 | → NORMAL |
| `false_alarm` | 标记误报 | 任意 | false_alarm | → 1 | 确认 | 标记误报：{note} | 误报：事件关闭 | → NORMAL |

**每次状态流转的副作用：**
1. 更新事件 `status` 和 `stageIndex`
2. 在 `logs` 数组**头部插入**一条新日志（`logs.unshift`）
3. 更新 `updatedAt` 为当前时间
4. 更新 `progressSummary`
5. 若 `resolve` 或 `false_alarm`：调用 `updateDeviceStatus(deviceId, NORMAL)` 恢复设备状态
6. 调用 `addNotification` 发送状态变更通知（resolve→success 类型，其余→info 类型）
7. 调用 `addPendingSync` 标记待同步

**`options` 参数：**
```javascript
{
  operator: "当前用户",   // 操作人，默认"当前用户"
  note: ""              // 备注，会写入日志内容和 progressSummary
}
```

## 5. 事件列表与筛选排序

### 5.1 事件列表数据源

事件列表来自 Pinia store `useRepairStore`（内部标识 `Jv`）：
- **全量事件** `e`：深拷贝自初始 Mock 数据 `Pwe`
- **活跃事件** `o`（`activeEvents`）：过滤掉 `resolved` 和 `false_alarm` 的事件
- **排序后事件** `l`（`sortedEvents`）：按规则排序的活跃事件，用于侧边栏列表展示

### 5.2 排序逻辑

`sortedEvents` 排序优先级（依次比较）：

1. **优先级 rank** 降序：`high(3) > medium(2) > low(1)`
2. **updatedAt** 升序：更新时间越早越靠前（即最新更新的排后面？实际看代码是 `x.updatedAt < E.updatedAt ? 1 : -1`，即越新的越靠后）
3. **stageIndex** 升序：阶段越靠前越优先
4. **source 权重**：`ai(1) < inspection(2) < crew(3)`

> 注意：排序结果中，高优先级、更新时间较旧的、处于早期阶段的事件排在前面，确保紧急且需要立即处理的事件优先展示。

### 5.3 筛选器

侧边栏提供三个筛选维度：

| 筛选器 | 字段 | 选项 |
|--------|------|------|
| 设备系统 | `c.device`（设备系统标识） | 主机系统 / 辅机系统 / 泵类设备 / 阀门管系 / 甲板机械 / 安全设备 |
| 优先级 | `c.priority` | high（高）/ medium（中）/ low（低） |
| 状态 | `c.status` | pending（待确认）/ processing（处理中）/ resolved（已解决） |

- 筛选条件为 `null` 时表示「全部」
- 提供「清空」按钮 `ct()`，重置所有筛选条件和搜索框

### 5.4 搜索

顶部搜索框支持模糊搜索事件 `title`、`deviceName`、`description` 三个字段。

### 5.5 事件卡片展示

每个事件卡片在侧边栏中显示：
- 优先级色条（左侧边框颜色）
- 事件标题
- 设备名称
- 来源标签（AI检测 / 巡检发现 / 船员上报）
- 状态标签（带对应颜色）
- 更新时间
- `progressSummary`（进度摘要）

## 6. 事件详情区：Tab 与展示

### 6.1 Tab 自动切换逻辑

Tab 的切换由 store 的 `selectEvent` 触发的 `f(id)` 函数决定：

```javascript
function f(id) {
  const event = events.find(e => e.id === id);
  if (!event) return;
  if (event.status === Ia.PENDING) {
    activeTab = "ai";           // 待确认 → AI分析Tab
  } else if (event.status === Ia.PROCESSING) {
    activeTab = "todo";         // 处理中 → 排查维修Tab
    // (stageIndex >= 2 也会进入 todo)
  } else if (event.status === Ia.TEMP_HANDLED) {
    activeTab = "log";          // 临时处理 → 操作日志Tab
  }
}
```

用户也可手动切换 Tab（调用 `setTab(tab)`）。

| 事件状态 | 默认 Tab | 说明 |
|---------|---------|------|
| pending | `ai`（AI分析） | 优先查看 AI 诊断结果以辅助确认 |
| processing | `todo`（排查维修） | 进入实际处置，执行排查/维修项 |
| temp_handled | `log`（操作日志） | 查看临时处理记录，决定后续 |
| resolved | 任意 | 已归档，各 Tab 均可查看 |
| false_alarm | 任意 | 已关闭，各 Tab 均可查看 |

### 6.2 事件头部信息

详情区顶部展示：
- 事件标题
- 设备名称 + 系统分类
- 优先级标签（颜色对应 high/medium/low）
- 来源标签
- 创建时间 / 更新时间
- `progressSummary` 进度摘要
- 状态操作按钮区（根据当前状态动态显示，见第 7 节）

### 6.3 AI 分析 Tab（`ai`）

展示完整的 AI 诊断信息：

1. **诊断结论区**：`verdict` 判定 + `confidence` 置信度 + `conclusion` 结论文字
2. **处置建议列表**：`suggestionsList` 数组逐条展示（编号列表）
3. **可能的原因与后果表格**：`faultTable` 渲染为三列表格（候选故障 / 可能原因 / 直接后果）
4. **数据分析卡片**：`dataCards` 以卡片网格展示，每张卡片包含：
   - 名称 `name`
   - 采样信息 `meta`
   - 实测值 `value`（支持 HTML 富文本，含 `<strong>` 高亮）
   - 判定 `verdict` + `verdictType` 颜色
   - 点击卡片可切换下方趋势图
5. **趋势曲线图**：基于 `trendData` 用 ECharts 渲染折线图，展示传感器数据随时间变化
6. **工程机理分析**（可选）：`engineering.intro`（HTML）+ `engineering.matches` 故障匹配度列表
7. **自动执行动作**（可选）：`autoActions` 列表

### 6.4 排查维修 Tab（`todo`）

展示两个子区块：

1. **排查项列表**（`todos.inspections`）：
   - 每个排查项可折叠/展开（`defaultFold` 控制初始状态）
   - 展开后显示 `steps` 步骤列表，每步含 `title` + `items`（检查清单）
   - 部分步骤标记 `hasImages: true` 表示需要拍照记录
   - 每个排查项底部有反馈区：选择「正常 / 异常」+ 填写详情 → 提交

2. **维修项列表**（`todos.repairs`）：
   - 结构同排查项
   - 反馈选项：「已解决 / 部分完成 / 临时处理」+ 详情

### 6.5 操作日志 Tab（`log`）

- 按 `logs` 数组倒序展示所有操作记录
- 每条日志显示：时间、阶段标签、标题、内容、操作人
- 阶段标签颜色区分：确认 / 排查 / 维修 / 归档

## 7. 操作动作详解

### 7.1 状态操作按钮

操作按钮根据事件当前状态动态显示：

| 当前状态 | 可用操作 |
|---------|---------|
| pending | ① 确认事件（confirm） ② 标记误报（false_alarm） |
| processing（stageIndex=1） | ① 进入维修（to_repair） ② 临时处理（temp_handle） ③ 标记误报 |
| processing（stageIndex=2） | ① 临时处理 ② 标记解决（resolve） ③ 标记误报 |
| temp_handled | ① 标记解决 ② 标记误报 |
| resolved | （无操作按钮，仅查看） |
| false_alarm | （无操作按钮，仅查看） |

所有状态下均可使用：**补充说明**（见第 9 节）。

### 7.2 确认事件（confirm）

- **前置状态**：pending
- **效果**：状态 pending → processing，stageIndex 0 → 1
- **日志**：`{stage: "确认", title: "事件已确认，进入排查", content: "事件已确认，进入排查"}`
- **progressSummary**：`"排查中：待执行排查项"`
- **Tab 切换**：自动切到 `todo`（排查维修）
- **用途**：值班人员确认 AI 检测/船员上报的异常为真实事件，正式启动处置流程

### 7.3 进入维修（to_repair）

- **前置状态**：processing（stageIndex=1，即已确认且在排查阶段）
- **效果**：stageIndex 1 → 2（状态保持 processing）
- **日志**：`{stage: "维修", title: "排查完成，进入维修阶段"}`
- **progressSummary**：`"维修中：待执行维修项"`
- **用途**：排查项执行完毕，确认需要维修，进入维修阶段

### 7.4 临时处理（temp_handle）

- **前置状态**：任意未关闭状态
- **效果**：状态 → temp_handled，stageIndex → 2
- **日志**：`{stage: "维修", title: "临时处理：{note||已临时处置}"}`
- **progressSummary**：`"临时处理：待彻底排查/复装"`
- **用途**：现场做了临时处置（如调高旁通阀开度），但未根本解决，需后续彻底处理

### 7.5 标记解决（resolve）

- **前置状态**：任意未关闭状态
- **效果**：状态 → resolved，stageIndex → 2
- **日志**：`{stage: "归档", title: "事件已解决并归档：{note||处理完成}"}`
- **progressSummary**：`"已归档：事件闭环"`
- **设备状态**：调用 `updateDeviceStatus(deviceId, NORMAL)`，设备恢复为正常
- **通知类型**：success
- **用途**：事件彻底解决，闭环归档

### 7.6 标记误报（false_alarm）

- **前置状态**：任意未关闭状态
- **效果**：状态 → false_alarm，stageIndex → 1
- **日志**：`{stage: "确认", title: "标记误报：{note||经核实为误报}"}`
- **progressSummary**：`"误报：事件关闭"`
- **设备状态**：调用 `updateDeviceStatus(deviceId, NORMAL)`，设备恢复为正常
- **用途**：经核实为传感器误报或非真实故障，关闭事件

## 8. 排查项与维修项反馈

### 8.1 排查项反馈（updateInspection）

```javascript
function updateInspection(eventId, inspectionId, result, detail, operator = "当前用户")
```

- **result**：`"normal"`（正常）或 `"abnormal"`（异常）
- **detail**：文字描述详情
- **效果**：
  - 更新对应排查项的 `result`、`detail`、`operator`、`time`（当前时间）
  - 设置 `defaultFold = true`（反馈后自动折叠）
  - 添加日志：`{stage: "排查", title: "排查项反馈：{inspection.title}", content: "判定{正常/异常}。{detail}"}`
  - 更新事件 `updatedAt`
  - 标记待同步

### 8.2 维修项反馈（updateRepair）

```javascript
function updateRepair(eventId, repairId, result, detail, operator = "当前用户")
```

- **result**：`"resolved"`（已解决）/ `"partial"`（部分完成）/ `"temp"`（临时处理）
- **detail**：文字描述详情
- **效果**：
  - 更新对应维修项的 `result`、`detail`、`operator`、`time`
  - 设置 `defaultFold = true`
  - 添加日志：`{stage: "维修", title: "维修项反馈：{repair.title}", content: "处理情况：{已解决/部分完成/临时处理}。{detail}"}`
  - 更新事件 `updatedAt`
  - 标记待同步

> 注意：维修项反馈**不会**自动改变事件状态。事件状态需通过第 7 节的状态操作按钮单独变更。

### 8.3 排查/维修步骤结构

每个排查项或维修项包含 `steps` 数组，每个步骤：

```javascript
{
  title: "检查液压油箱外观及液位计",   // 步骤标题
  items: [                            // 检查清单（字符串数组）
    "目视检查油箱外壁焊缝处有无渗漏油渍",
    "检查油箱底部放油堵有无湿润痕迹",
    "对比液位计读数与远程监控值是否一致",
    "检查液位计上下连通阀是否处于全开位置"
  ],
  time: "",                            // 步骤执行时间记录
  hasImages: false                     // 是否需要拍照（true 时展示图片上传区）
}
```

## 10. 事件创建与归并

### 10.1 创建事件（createEvent）

```javascript
function createEvent({ deviceId, deviceName, systemPart, title, description,
                       priority = "medium", source = "crew", sensorId })
```

**归并检测**：创建前先查找是否存在同设备（`deviceId`）、同标题（`title`）、且未关闭（非 resolved/false_alarm）的事件：
- **存在** → 归并：添加日志 `{stage: "确认", title: "重复报警归并", content: "同设备同现象重复上报，归并至本事件：{description}"}`，发送归并通知，返回已有事件
- **不存在** → 创建新事件

**新事件初始化**：
```javascript
{
  id: `EVT-2026-${String(43 + events.length).padStart(4, "0")}`,  // 自增ID
  title, systemPart, deviceId, deviceName, priority, source,
  status: Ia.PENDING,        // 待确认
  stageIndex: 0,             // 确认阶段
  createdAt: now(),
  updatedAt: now(),
  description,
  progressSummary: "待确认：船员上报，待人工确认",
  aiAnalysis: {
    conclusion: "初步AI分析中：根据上报信息匹配历史案例，生成排查建议",
    causes: [{ cause: "待排查确认", consequence: "待评估", probability: "待定" }],
    basis: "基于上报描述的初步推断",
    snapshot: sensorId ? [{ name: "上报传感器", value: "-", unit: "", range: [0,0], status: "warning" }] : []
  },
  todos: {
    inspections: [{ id: "T1", title: "现场核实异常现象", defaultFold: false, result: null, detail: "", operator: "", time: "", steps: [...] }],
    repairs: []
  },
  logs: [{ time: now(), stage: "确认", title: "船员上报异常", content: description, operator: "当前用户" }]
}
```

### 10.2 事件来源

| 来源 | 触发场景 | priority 默认 | 初始 progressSummary |
|------|---------|--------------|---------------------|
| `ai` / `ai_detection` | 监控中心传感器 z-score 超阈值，多传感器联合偏离检测 | 按 AI 判定 | 待确认：AI检测触发，待人工确认 |
| `inspection` | 机舱巡检日检报告中发现异常 | medium | 待确认：巡检发现，待人工确认 |
| `crew` / `crew_report` | 船员通过异常上报弹窗手动提交 | medium | 待确认：船员上报，待人工确认 |

## 11. AI 分析模块

### 11.1 AI 分析数据结构

AI 分析结果存储在 `event.aiAnalysis` 中，包含以下子模块：

#### 11.1.1 诊断判定（verdict + confidence）

| verdict 值 | 含义 | 说明 |
|-----------|------|------|
| `suspected` | 疑似故障 | AI 初步判断存在故障，置信度反映把握程度 |
| `confirmed` | 确认故障 | 经人工或检测确认的故障 |
| `resolved` | 已解决 | 故障已处理 |

`confidence` 为 0-100 的整数，表示 AI 对诊断结论的置信度。

#### 11.1.2 可能原因表（faultTable）

```javascript
faultTable: [
  {
    candidate: "管路渗漏",       // 候选故障名称
    reason: "...",              // 可能原因分析
    consequence: "...",         // 直接后果
    probability: "高"           // 发生概率: 高 | 中 | 低
  }
]
```

概率颜色映射：高 → `#FF4D4F`（红），中 → `#FAAD14`（橙），低 → `#52C41A`（绿）。

#### 11.1.3 数据分析卡片（dataCards）

```javascript
dataCards: [
  {
    name: "P43-液压油位（核心监控）",  // 指标名称
    meta: "20个采样点 · 约20h趋势 · 正常范围 30~100%",  // 采样信息
    value: "当前 <strong>18%</strong>，20h前 <strong>98%</strong>...",  // 实测值（含HTML）
    verdict: "严重偏低 — 存在泵吸空风险",   // 判定文字
    verdictType: "danger"                    // 判定类型: normal | warning | danger
  }
]
```

`verdictType` 颜色映射：
- `normal` → `#52C41A`（绿，正常）
- `warning` → `#FAAD14`（橙，警告）
- `danger` → `#FF4D4F`（红，异常）

#### 11.1.4 趋势数据（trendData）

```javascript
trendData: {
  seriesName: "液压油位",       // 曲线名称
  yUnit: "%",                   // Y轴单位
  yMin: 10,                     // Y轴最小值
  yMax: 105,                    // Y轴最大值
  times: ["06-22 00:00", "06-22 04:00", ...],  // 时间轴
  values: [98, 95, 92, ...]     // 对应数值
}
```

#### 11.1.5 工程机理分析（engineering）

```javascript
engineering: {
  intro: "<p>...</p>",          // HTML格式原理介绍
  matches: [                    // 故障匹配度评估
    {
      fault: "管路外部渗漏",
      match: "高匹配",          // 高匹配 | 部分匹配 | 低匹配 | 待排除
      cls: "high",              // high | medium | low | pending
      reason: "..."
    }
  ]
}
```

#### 11.1.6 同类案例（relatedCases）

```javascript
relatedCases: [
  {
    title: "离心泵检修后振动偏高·联轴器对中偏差",
    match: 85,                  // 匹配度 0-100
    solution: "激光对中仪重新找正，径向偏差降至0.05mm以内..."
  }
]
```

#### 11.1.7 处置建议（suggestionsList）

字符串数组，每条为一个完整的建议步骤：

```javascript
suggestionsList: [
  "立即补充液压油至正常油位（≥60%），优先选用 ISO VG 46 抗磨液压油",
  "对液压管路进行全面目视检查，重点检查各接头、法兰、软管段的渗漏痕迹",
  "使用紫外荧光检漏剂辅助定位微小泄漏点",
  ...
]
```

### 11.2 AI 分析生成时机

AI 分析在事件创建时自动生成，内容包含：
- `conclusionDetail`：Markdown 格式的详细分析报告（含 `**加粗**`、列表、段落）
- `autoActions`：记录系统自动执行的动作（如"自动创建排查任务"、"自动关联传感器数据"、"AI分析振动数据属优良范围"等）

## 12. AI 助手对话能力

检修中心事件详情区内置 AI 助手对话框，基于**关键词正则匹配**返回结构化回复。

### 12.1 对话触发逻辑

AI 助手根据用户输入文本 `Z`（已转为字符串）依次匹配以下正则，命中即返回对应结构化回复：

| 匹配正则 | 命中意图 | 返回内容 |
|---------|---------|---------|
| `/原因\|为什么\|什么问题\|什么故障\|怎么回事\|分析\|是不是\|是主因\|导致\|引起的/` | 原因分析 | `faultTable` 完整列表（含概率着色）+ 诊断结论 + 置信度 + 设备名 |
| `/趋势\|数据\|传感器\|读数\|多少\|当前值/` | 传感器数据 | `dataCards` 完整列表（含采样信息、实测数据、判定）+ 趋势曲线数据 |
| `/检查\|排查\|处置\|怎么修\|如何\|步骤\|怎么办/` | 排查步骤 | `todos.inspections[0]` 的前5步（含子项）+ 步骤总数 |
| `/处置建议\|怎么处理\|解决方案\|建议\|怎么办\|措施/` | 处置建议 | `suggestionsList` 完整列表 + 诊断结论 |
| `/案例\|同类\|历史故障\|相似\|以前.*发生/` | 同类案例 | `relatedCases` 完整列表（含匹配度和解决方案） |
| （默认，未命中上述） | 事件概览 | 事件标题 + 描述 + 可能原因前2项 + 提示可继续提问的方向 |

### 12.2 默认回复格式

未命中任何关键词时，返回事件概览：
- 事件标题 + 描述
- AI 诊断结论
- 可能原因（前2项，超出显示"…"）
- 核心传感器数据（如有）
- 提示语：「可继续问我：原因分析 / 传感器数据 / 同类案例 / 处置建议」

### 12.3 回复内容格式

所有回复均为 HTML 富文本，使用 `<br>`、`<strong>`、`<span style="color:...">` 等标签：
- 概率着色：高→`#FF4D4F`，中→`#FAAD14`，低→`#52C41A`
- 判定着色：danger→`#FF4D4F`，warning→`#FAAD14`，normal→`#52C41A`
- 提示语着色：`#8899b0`（灰蓝，辅助说明）
- 引导词着色：`#00bcd4`（青色，可点击的引导）

### 12.4 快捷回复

AI 助手提供快捷回复按钮（quickReply），用户可点击直接发送预设问题：
- 原因分析
- 传感器数据
- 同类案例
- 处置建议

## 13. 通知与同步机制

### 13.1 通知系统

通知存储在全局 store 中，每条通知结构见 [3.3 节](#33-通知对象结构)。

**通知触发时机：**
- 新事件创建：`{type: "danger", title: "新事件：{title}", kind: "new"}`
- 状态变更：`{type: "info"|"success", title: "状态变更：{title} → {新状态标签}", kind: "change"}`
- 重复报警归并：`{type: "info", title: "重复报警已归并：{title}", kind: "new"}`

**通知颜色映射：**
- `danger` → `#FF4D4F`（红）
- `warning` → `#FAAD14`（橙）
- `info` → `#1890FF`（蓝）
- `success` → `#52C41A`（绿）

### 13.2 待同步标记

每次事件状态变更、排查/维修项反馈、补充说明提交后，都会调用 `addPendingSync()` 标记数据有变更待同步。这用于：
- 模拟离线数据同步机制
- 提示用户有未同步的本地修改

### 13.3 日志系统

每次操作都会在事件的 `logs` 数组**头部插入**（`unshift`）一条记录，确保最新操作显示在最前面。日志的 `stage` 字段对应操作所属阶段：

| stage 值 | 含义 | 典型操作 |
|---------|------|---------|
| 确认 | 事件确认阶段 | 创建事件、确认事件、标记误报、补充说明 |
| 排查 | 排查阶段 | 排查项反馈 |
| 维修 | 维修阶段 | 进入维修、临时处理、维修项反馈 |
| 归档 | 归档阶段 | 标记解决 |
