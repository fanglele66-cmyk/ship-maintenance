# 船舶设备检修助手 V2.0 改造需求文档

> **版本**：V2.0  
> **日期**：2026-07-03  
> **适用场景**：船端 pad 高频走动巡检  
> **改造分支**：`feat/event-center-v2`（原 `main` 分支保留不动）  
> **原型参考**：`V2基础原型.html` / `事件中心V2完整架构框架.html` / `concept-v2.html`

---

## 一、改造背景与目标

### 1.1 现状问题

V1.0 版本采用**信息地图式**布局：首页仪表盘 + 监控中心设备树 + 检修工单列表。所有信息平铺在一个页面，用户打开后需要自己找信息、判断问题、决定操作。

**核心痛点：**
- 信息密度过高，用户一眼看不清重点
- 缺乏引导，用户不知道"接下来该干什么"
- 交互层级偏平，没有从宏观到微观的递进感
- AI 助手作为辅助工具存在，但不是交互核心

### 1.2 改造目标

将产品从**静态信息看板**升级为**事件驱动 + AI 引导**的智能交互系统。

| 维度 | V1.0（现状） | V2.0（目标） |
|------|-------------|-------------|
| 交互模式 | 用户找信息 | 系统推送+AI引导 |
| 信息组织 | 所有数据平铺 | 按事件聚合，按层级递进 |
| 助手定位 | 辅助工具 | 主工作区/智能副驾驶 |
| 适用场景 | 坐着看 | 走动巡检 |

### 1.3 用户画像

- **使用者**：船上轮机员/机工
- **使用设备**：pad（横屏）
- **使用场景**：手持 pad 在机舱走动巡检，高频使用
- **操作特点**：可能单手操作，注意力碎片化，需要快速响应
- **网络环境**：仅考虑在线模式（不考虑离线场景）

### 1.4 设计原则

1. **事件驱动**：系统主动推送，用户不找信息
2. **助手主导**：事件中心所有交互操作在助手对话中完成
3. **层级递进**：态势感知从宏观到微观逐层深入
4. **统一框架**：核心 Tab 采用一致的布局（左侧信息 + 右侧助手）
5. **快照 vs 实时**：事件中心 = 冻结快照，态势感知 = 定时刷新实时
6. **视觉延续**：态势感知严格沿用 V1.0 首页和监控中心的视觉元素，不重新设计

## 二、整体架构变更

### 2.1 Tab 架构变更

| V1.0 Tab | V2.0 Tab | 变更说明 |
|-----------|-----------|---------|
| 首页 | ❌ 移除 | 首页功能融合到"态势感知" |
| 监控中心 | ⬅️ 改名为"态势感知" | 与原首页功能合并 |
| 检修中心 | ⬅️ 改名为"事件中心" | 从工单列表改为事件驱动流 |
| — | 📖 知识库 | **沿用 V1.0 数据和布局**，仅入口从顶栏移到导航 |
| 个人中心 | ❌ 移除 | 一船一 pad 共享，不需要个人中心 |

> **重要说明**：知识库不是"新增"功能，而是 V1.0 已有的功能。V2.0 中知识库的数据（8条）、布局（分类标签+列表）、功能（搜索+详情）全部保持不变，唯一的变更是将入口从 V1.0 顶栏移到了 V2.0 左侧导航栏。

### 2.2 导航栏变更

| V1.0 | V2.0 |
|------|------|
| 左侧可折叠边栏（200px↔64px），文字+图标 | 最左侧窄导航栏（56px），图标+小字，固定不可折叠 |
| 导航项：首页/监控/检修 | 导航项：事件中心/态势感知/知识库 |
| 顶部还有：船舶名/网络/通知/知识库/AI助手/异常上报 | 顶部仅保留：船舶名+航行状态/网络指示/实时时间。通知和异常上报由事件中心和助手接管 |

**V2.0 导航栏视觉规格：**
- 宽度：56px（固定）
- 背景：`#0C1A2E`
- 每个导航项：图标(18px) + 文字(9px)，垂直排列
- 选中态：背景 `rgba(24,144,255,0.15)`，文字/图标 `#1890FF`
- Hover 态：背景 `rgba(24,144,255,0.08)`，文字 `#8BAAC0`

### 2.3 顶栏变更（AppHeader 简化）

| V1.0 顶栏元素 | V2.0 处理 |
|---------------|-----------|
| 船舶名 + 航行状态标签 | ✅ 保留 |
| 实时时间 | ✅ 保留 |
| 网络指示（NetworkIndicator） | ✅ 保留，但仅在线模式 |
| 通知铃铛（带角标下拉） | ❌ 移除，通知由事件中心事件流接管 |
| 知识库入口 | ❌ 移除，改为左侧导航独立 Tab |
| AI助手入口 | ❌ 移除，助手改为各 Tab 内常驻 |
| 异常上报按钮（红色） | ❌ 移除，上报通过助手对话完成 |

### 2.4 三个 Tab 的布局统一

所有核心 Tab 采用**左侧信息区 + 右侧常驻助手**的统一框架：

```
┌──────┬──────────────────────┬───────────
│ 导航  │   左侧信息区(flex)     │ AI助手    │
│ 56px │                      │ 300px    │
│      │                      │ (常驻)    │
└────────────────────────────┴───────────
```

知识库 Tab 例外：无助手，纯浏览布局。

## 三、事件中心（Tab 1 · 核心，默认进入）

### 3.1 定位

事件中心是用户 **80% 时间停留**的地方。核心理念：**不是用户找问题，而是系统告诉用户该处理什么。**

### 3.2 数据模型

- **事件 = 异常告警 + 工单 + 维保提醒的统一聚合体**
- 每个事件携带**快照数据**：事件发生瞬间的传感器数据被冻结保存，不会随时间变化
- 快照是"事发时的证据"，用于事后追溯

**事件数据结构：**

```javascript
{
  id: 'EVT-2026-001',
  title: '主机冷却水温异常偏高',
  system: '主机系统',
  priority: 'critical',      // critical / important / normal
  status: 'pending',          // pending / processing / resolved
  createdAt: '2026-07-02T14:05:00',
  source: 'sensor_alarm',     // sensor_alarm / manual_report / maintenance_schedule / situation_referral
  
  // 快照数据（事发瞬间冻结）
  snapshot: {
    sensors: [
      { name: '冷却水出口温度', value: 92.4, unit: '°C', threshold: 85, status: 'over' },
      { name: '冷却水进口温度', value: 48.7, unit: '°C', threshold: 55, status: 'normal' },
      { name: '温差', value: 43.7, unit: '°C', range: '15-25', status: 'over' },
      // ... 6 个关键指标
    ],
    trendData: [/* 24h 数据点数组 */],
    snapshotTime: '2026-07-02T14:05:00'
  },
  
  // AI 诊断
  aiAnalysis: {
    summary: '冷却水温差异常偏大（43.7°C），高度怀疑冷却系统存在堵塞或热交换器效率下降。',
    faultTable: [
      { name: '海水冷却管路堵塞', probability: 'high', detail: '...' },
      { name: '淡水冷却侧水垢', probability: 'medium', detail: '...' },
      { name: '温控阀故障', probability: 'low', detail: '...' }
    ],
    suggestions: '建议优先检查：① 海水冷却管路 ② 淡水冷却侧水垢 ③ 温控阀工作状态'
  },
  
  // 关联案例
  relatedCases: [
    { title: '2026-05 主机冷却水系统堵塞案例', url: '...' }
  ],
  
  // 处理时间线
  timeline: [
    { time: '2026-07-02T14:05:00', action: '事件创建（系统自动）' },
    { time: '2026-07-02T14:10:00', action: '用户查看快照' },
    // ...
  ]
}
```

### 3.3 布局与交互

#### 3.3.1 默认状态（浏览态）

```
┌──────┬──────────────┬───────────
│ 导航  │  事件列表     │ AI助手    │
│      │  280px        │ (通用模式) │
│      │              │           │
│      │ • 事件卡片1   │ 欢迎语    │
│      │ • 事件卡片2   │ 总览摘要  │
│      │ • 事件卡片3   │ 推荐问题  │
│      │              │           │
└──────┴──────────────┴───────────
```

**事件列表（左侧 280px）：**

- 顶部 Header：
  - 标题"事件中心" + 事件总数
  - 筛选标签栏：全部 / 待处理 / 处理中 / 已解决
- 每条事件卡片：
  - 左侧色点： 紧急(critical) / 🟡 预警(warning) / 🔵 信息(info)
  - 事件标题（单行省略）
  - 所属系统 + 优先级
  - 发生时间
  - 拖拽手柄（⠿ 图标，hover 显示）
- 支持**拖拽排序**（HTML5 Drag API 或 SortableJS）
- 默认按时间倒序排列，最新的在最上面
- 点击事件卡片 → 进入聚焦态

**事件卡片视觉规格：**
- 背景：`#0B1A2E`
- 边框：底部 `1px solid #0F1F38`
- Hover：背景 `rgba(24,144,255,0.05)`
- 选中：背景 `rgba(24,144,255,0.1)` + 左边框 `2px solid #1890FF`
- 内边距：`10px 14px`

**助手（右侧 300px，通用模式）：**

- 欢迎语（纯文本 `greetingText`）：如"你好，当前共有 6 件事件待处理"
- 总览摘要（HTML 富文本 `summaryHtml`，`v-html` 渲染）：
  - 船舶健康分
  - 事件统计（紧急 X / 预警 X / 信息 X）
  - 重点关注建议
- 3 个推荐问题 Chips（总览级别）：
  - 示例："今天有几件紧急事件？" / "最近哪个系统问题最多？" / "有没有重复出现的故障？"
- 点击 Chip 后该 Chip 变灰绿+✓ 标记但**保持可见**，用户可连续点击多个
- 输入框 +  语音输入按钮

#### 3.3.2 聚焦状态（选中事件后）

```
┌──────┬──────────────┬───────────
│ 导航  │  产物抽屉     │ AI助手    │
│      │  (从左滑入)   │ (事件模式) │
│      │              │           │
│      │ 快照数据      │ 事件分析  │
│      │ 趋势图        │ 推荐操作  │
│      │ AI诊断        │ 操作卡片  │
│      │              │ 推荐问题  │
└──────┴──────────────┴───────────
     ↑ 左栏事件列表丝滑收起(width→0)
```

**交互流程（步骤级）：**

1. 用户点击事件卡片
2. 事件列表 CSS 过渡：`width: 280px → 0`, `opacity: 1 → 0`, `border-right: none`，`transition: 0.3s ease`
3. 产物抽屉 CSS 过渡：`display: flex`, `flex: 1`，从左侧滑入
4. 助手自动从"通用模式"切换到"事件模式"
5. 助手调用 `pushEventContextIfNeed(eventId)` 推送事件上下文
6. 返回操作：点击产物抽屉左上角 ← 箭头 → 产物抽屉消失 → 事件列表重新展开

**产物抽屉（只读，不交互）：**

- Header：
  - 返回箭头 ←
  - 事件标题
  - 状态标签（色块）
- 事件快照数据区：
  - 标题"事件快照 · 事发时数据"
  - 网格卡片（3列），每张卡片：指标名 + 冻结值（大字号 Consolas）+ 单位
  - 超阈值：值红色 `#FF4D4F`
  - 预警：值黄色 `#FAAD14`
  - 正常：值白色 `#E8F0FF`
- 趋势图区：
  - 标题"温度趋势(事发前 24h)"
  - ECharts 折线图：数据曲线 + 阈值虚线 + 突变点圆点标注 + 渐变面积填充
- AI 诊断分析区：
  - 标题"AI 诊断分析"
  - 渐变背景卡片（`rgba(24,144,255,0.08) → rgba(0,188,212,0.08)`）
  - AI 初步判断文本
  - 可能原因列表
  - 优先排查建议
- **底部提示文案**：`"如需操作，请联系右侧 AI 助手"`（灰色小字居中）
- **产物区无任何操作按钮**，纯展示

**助手（事件模式）：**

- 自动推送该事件的上下文消息：
  ```
  🤖 事件已关联：主机冷却水温异常偏高
  
  快照数据显示冷却水出口温度 92.4°C，超阈值 85°C。
  AI 判断可能是冷却系统堵塞或热交换器效率下降。
  
  我可以帮你：
  [🔍 查看可能原因] [📋 开始排查] [📝 创建工单]
  
  ┌─────────────────────────────┐
  │ 推荐操作                      │
  │ 根据 AI 分析，建议先排查       │
  │ 海水冷却管路是否堵塞。         │
  │ [开始排查] [转派工单] [标记误报]│
  └─────────────────────────────┘
  ```
- 推荐问题 Chips（2-3 个，根据事件数据动态生成）
- 操作卡片（嵌入在对话气泡内）：标题 + 简述 + 操作按钮组
- 所有事件操作都在助手对话中完成
- 支持事件切换：助手顶部有下拉选择器，可自由切换关联事件
- 事件切换时自动切换到该事件的独立会话

#### 3.3.3 助手事件模式 · 推荐问题差异化规则

每个事件的推荐问题基于该事件自身的 `aiAnalysis` 数据动态生成：

| 数据来源 | 生成的问题示例 |
|---------|--------------|
| faultTable 候选原因 | "查看可能的原因" / "第二个可能原因是什么？" |
| sensorCode 异常传感器 | "Exh.Gas Temp 为什么会超限？" / "这个传感器正常范围是多少？" |
| relatedCases 关联案例 | "2026-05 那次堵塞是怎么处理的？" |
| 通用排查 | "开始排查步骤" / "需要准备什么工具？" |
| 工单操作 | "创建工单" / "转派给谁？" / "标记为误报" |

#### 3.3.4 助手事件模式 · genReply 回复引擎分支

助手需要识别用户自然语言，路由到不同回复分支：

| 关键词/意图 | 回复分支 | 回复内容 |
|------------|---------|---------|
| 查看/看看/详情 | 事件概览 | 展示事件快照数据摘要 + 趋势描述 |
| 原因/为什么/怎么回事 | 故障原因 | 列出 faultTable 中的候选原因，按概率排序 |
| 传感器/温度/压力/数值 | 传感器解读 | 解释该传感器的当前值、阈值、正常范围 |
| 排查/检查/怎么办 | 排查步骤 | 推送 SOP 步骤卡片（来源：知识库匹配） |
| 建议/推荐/操作 | 操作建议 | 推送操作卡片（开始排查/转派/误报） |
| 案例/上次/历史 | 关联案例 | 引用 relatedCases，展示历史处理经验 |
| 时间线/记录/进度 | 时间线 | 展示事件处理时间线（发现→排查→处理→归档） |

### 3.4 事件生命周期

```
异常告警产生
  → 系统创建事件(冻结快照)
  → 事件出现在列表(待处理 pending)
  → 用户点击 → 助手引导排查(处理中 processing)
  → 用户在助手中操作 → 创建/推进工单
  → 标记解决(已解决 resolved)
  → 事件沉到列表底部
```

工单三态保持 V1.0 约定：`pending → processing → resolved`

### 3.5 事件来源

| 来源 | 触发条件 | 示例 |
|------|---------|------|
| 传感器自动告警 | 数值超阈值 | 冷却水温 > 85°C → 自动冻结快照 → 生成事件 |
| 人工上报 | 用户通过助手描述 | "3号泵有异响" → 助手引导创建事件 |
| 维保计划到期 | 定时触发 | 3号泵500小时保养到期 |
| 态势感知跳转 | 用户主动触发 | 在态势感知发现异常 → 一键生成快照事件 |

### 3.6 事件列表筛选

| 筛选标签 | 显示规则 |
|---------|---------|
| 全部（默认） | 显示所有事件 |
| 待处理 | `status === 'pending'` |
| 处理中 | `status === 'processing'` |
| 已解决 | `status === 'resolved'`，沉到底部 |

## 四、态势感知（Tab 2）

> **视觉原则：态势感知 = V1.0 首页（船体线框图）+ V1.0 监控中心（传感器卡片+趋势图）的拼接，加上右侧常驻助手。视觉元素、卡片样式、图表形式严格沿用 V1.0 原版，不重新设计。**

### 4.1 定位

用户**主动来看数据**的地方。"现在船上情况怎么样？"的答案在这里。

与事件中心的本质区别：

| | 事件中心 | 态势感知 |
|---|---|---|
| 谁主动 | 系统推送给用户 | 用户主动来看 |
| 数据类型 | 快照（冻结） | 实时（定时刷新） |
| 助手角色 | 主导操作 | 辅助解读 |
| 用户目的 | "出了什么事，我该怎么办" | "现在船上情况怎么样" |

### 4.2 数据刷新机制

- **态势感知数据定时推送**，约每 30 分钟刷新一次
- 在两次刷新之间，如果出现异常预警，系统自动：
  1. 冻结当前异常数据为**快照**
  2. 将快照推送到**事件中心**的事件列表中
  3. 态势感知侧的数据保持上次刷新值不变，直到下次定时刷新

### 4.3 数据模型

- **船舶级**有健康分（0-100 分），如：82 分（良好）
- **设备级无健康分**，只有状态枚举：`normal / running / warning / danger / offline`
- 船舶信息：航行状态（sailing/anchoring/docking）+ 经纬度 + 航速 + 航向
- 设备信息：设备名 + 所属系统 + 状态 + 2 个关键运行指标

### 4.4 布局与交互

#### 4.4.1 整体布局

```
┌──────┬──────────────────────┬───────────
│ 导航  │  数据展示区(flex)     │ AI助手    │
│ 56px │  (三层递进，滑动切换)  │ 300px    │
│      │                      │ (常驻)    │
└──────┴──────────────────────┴───────────
```

助手与事件中心布局一致（300px 常驻），但行为模式为"辅助解读"。

#### 4.4.2 第一层：船舶线框图 + 悬浮设备卡片

// touchstart 记录起始坐标
// touchend 计算位移
// 水平位移 > 50px && |dx| > |dy| → 触发层级切换
// 左滑(dx<0) = 下一层，右滑(dx>0) = 上一层
```

**CSS 过渡：**

.layer {

```

### 4.5 助手在态势感知中的行为

#### 4.5.1 角色定义：数据参谋（辅助模式）

与事件中心的"驾驶员"角色不同，态势感知的助手是**静默跟随、按需解读**的配角。

| | 事件中心助手 | 态势感知助手 |
|---|---|---|
| 角色 | 驾驶员 | 数据参谋 |
| 主动性 | 高，主动推送操作卡片 | 低，静默跟随不打断 |
| 交互方式 | 对话内嵌按钮主导操作 | 用户点击数据卡片"问助手"触发解读 |
| 上下文 | 绑定当前事件快照 | 跟随用户浏览路径动态切换 |
| 特殊能力 | 驱动工单闭环 | 意图识别 → 反向引导（推荐知识/案例） |
| 衔接 | 接收异常快照 | 发现异常 → 一键生成事件快照 → 跳转事件中心 |

#### 4.5.2 助手跟随规则

| 用户操作 | 助手响应 |
|---------|---------|
| 进入态势感知（第一层） | 推送船舶概览：健康分 + 设备异常摘要 + 重点关注建议 |
| 点击设备卡片（进入第二层） | 推送该系统传感器概况：几个超限/预警，优先关注哪个 |
| 点击传感器卡片（进入第三层） | 推送趋势解读：最近变化 + 可能原因 + 是否建议生成事件快照 |
| 反复查看某设备（意图识别） | 主动推送关联知识/历史案例（"上次这台设备出过类似问题"） |
| 静默浏览 | 不打扰，不刷屏 |

#### 4.5.3 助手能力清单

| 能力 | 说明 | 示例 |
|------|------|------|
| **口语化查询** | 用户直接问数据，不用翻设备树 | "哪个泵温度最高？" → 助手返回排序结果 |
| **"问助手"入口** | 数据卡片旁有小图标，一键解读 | 趋势图旁点"AI解读" → 助手标注异常区间 |
| **异常快照跳转** | 发现异常 → 一键生成快照事件 | "这个温度不对，帮我上报" → 生成快照 → 跳转事件中心 |
| **关联回看** | 看设备时自动关联历史事件 | "上次这样出过问题：2026-05堵塞事件" |
| **AI 预测预警** | 数据趋势不对但还没超阈值 | "注意：排气温度上升速率加快，预计2小时内超限" |
| **语音输入** | 🎤 按钮，Web Speech API | 按住说话："主机温度多少？" → 助手回答 |

#### 4.5.4 助手推荐问题规则（跟随浏览位置）

| 浏览层级 | 推荐问题示例 |
|---------|------------|
| 第一层（船体图） | "哪些设备需要关注？" / "健康分为什么是82？" / "有异常的设备汇总" |
| 第二层（传感器网格） | "排气温度为什么会超限？" / "哪个传感器最危险？" / "相关排查步骤" |
| 第三层（趋势图） | "19:00的突变是什么原因？" / "生成事件快照" / "对比正常时期的趋势" |

- 识别结果自动填入输入框
- 支持中文语音识别

- relatedCases 关联案例列表
